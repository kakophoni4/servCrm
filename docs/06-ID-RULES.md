# Генерация ID заявки

## Решение дня 0

В исходном ТЗ указано «8 цифр», но состав полей:

```
MM (2) + YY (2) + W (1) + NNNN (4) = 9 цифр
```

**Выбрано: вариант C — 9 цифр.**  
Формат сохраняет неделю и 4 порядковых (до 9999 заявок/неделю), что критичнее, чем литерал «8».

Альтернативы (отклонены пока заказчик не скажет иное):

| Вариант | Формат | Длина |
|---------|--------|-------|
| A | MM + YY + W + NNN | 8 |
| B | MM + YY + NNNN | 8 |
| **C (принято)** | **MM + YY + W + NNNN** | **9** |

## Формат

```
MM YY W NNNN  →  MMYYWNNNN
```

| Часть | Длина | Описание |
|-------|-------|----------|
| MM | 2 | месяц, с ведущим нулём (01–12) |
| YY | 2 | год (26 = 2026) |
| W | 1 | номер недели месяца (1–5), см. ниже |
| NNNN | 4 | порядковый в рамках MM+YY+W, с ведущими нулями |

## Пример

Июль 2026, неделя 3, порядковый 0042 → **`072634042`**

## Неделя месяца

Неделя месяца = `ceil(day_of_month / 7)`, диапазон 1–5.

Примеры для июля:

- 1–7 → W=1
- 8–14 → W=2
- 15–21 → W=3
- 22–28 → W=4
- 29–31 → W=5

База даты для расчёта: **дата создания заявки** (`created_at`, локаль филиала / Europe/Moscow).

## Уникальность и генерация

1. Вычислить префикс `MMYYW`.
2. В транзакции найти `MAX(seq)` для этого префикса, `seq + 1`.
3. Собрать ID = `prefix + pad(seq, 4)`.
4. При коллизии — retry.
5. ID **не меняется** при смене статуса / мастера / дате выполнения.

Хранение: строка (или bigint); отображение — как есть, без разделителей.
